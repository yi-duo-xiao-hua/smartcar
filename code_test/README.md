### 开发历史简介  
这里简单介绍一下我们小车代码的开发历史和迭代过程，方便理解接下来下面各个代码文件以及其用途。  
**1. 初始阶段**  
这个阶段主要是初步完成比赛要求的各个任务，包括二维码识别、大模型通信、小车PID控制等等，文件一般以test开头进行命名，其中二维码识别、大模型的几个代码直到最后也基本没有大改动，因此一直保留着。而涉及到小车控制的代码，包括Resnet和Yolo的代码，因为控制逻辑经过多次迭代，这个时候的这些代码基本都被弃用了，只是作为留存而保留在项目中。这部分代码主要由陈柏宏开发。  
最初的避障时PID的计算可以参考我们的代码，在我的Markdown里没有太多提及。  
**2. 第一次控制程序迭代**  
夏学期考试周结束后我接手了代码编写的工作，期间我尝试了一些针对原先避障的改善方案。包括引入三种巡线路径以减小避障的转向幅度，但是因为三条路径的自动切换始终存在问题，所以这个方案被放弃了。还有进入避障的判断采用小车前进区域和障碍物矩形框是否相交，虽然这个避障判断方案有一些问题，也被放弃了，但是为后续的避障方法迭代奠定了基础。  
第一次重要的控制逻辑迭代，引入了小车前进区域的两条直线以及对应的避障逻辑，这之后新编写的控制代码基本以new开头。迭代后的控制程序基本还是和原来的控制程序类似，只是避障的PID计算不同。  
**3. 第二次控制程序迭代**  
第二次重要的控制逻辑迭代，算是一次巨大的结构性改动，PID的控制计算从原来的Resnet节点移动到了Yolo节点，Resnet节点变为了只承担调用模型识别注意点并将结果发布。而Yolo节点则承接了PID控制的任务，同时，对于Camera的Yolo模型识别结果的解析也改变了，原来只解析出面积最大的障碍物将其发给Resnet节点，现在Yolo节点会解析所有的识别结果，这为识别P点、二维码带来了基础，同时，也给后面可能结合多个障碍物避障甚至进行路径规划打基础，虽然到最后都只是针对最大面积的障碍物进行避障。  
**4. 小迭代**  
第二次重要迭代之后程序的逻辑就基本确定了，接下来的都是一些小迭代。  
在学会使用深度相机后，我们尝试了通过深度相机获取障碍物的距离，取代原先使用的面积，但是因为深度相机和单目摄像机的安装位置不同存在视角差异，最终这个方案被放弃了。深度相机最后只被用来识别二维码和图生文。  
在修改了STM32代码后，许多PID参数需要重新调整，为此，我对代码进行了一些改进，加入了读取配置文件的功能，这样将需要修改的量写入配置文件，就避免了重新编译带来的麻烦。  
最后，针对一些刁钻的障碍物摆放，我又重新引入了原先的三条路径的策略，只是这次我选择了路径策略写在配置文件中，虽然这样是写死了路径策略，但是避免了繁琐的小车自主切换路径策略。

### 小车功能包(`test_carcontrol`)  
***包含下面代码文件：***

**其中以下是实际比赛时使用的：**
* `image_comp.py`：压缩图像程序，订阅单目摄像机的图像话题，将其按照一定的比例如1/2进行压缩，从而减小向上位机传输的数据量。国赛时临时编写的，在我们自己测试时，在网络带宽受限的情况下可以保持视频流畅，但是在国赛实际比赛时网络拥堵的情况下效果并不好。
* `new_ctrlpub.py`：省赛前临时写的程序，用于向/sign4return发布数据，主要是给队员A使用，可以控制小车进入自动驾驶状态或者停下。
* `new_resnet.py`：Resnet节点代码，用于调用Resnet模型识别注意点，将结果发布到对应话题。
* `new_yolo_d.py`：Yolo节点代码。最初是将使用深度相机得到的距离替代原先的障碍物面积时写的，后来这个方案被弃用了。这个节点代码成为了用于测试和更新的代码，所有新的功能或者逻辑都会先写在这个程序里，等通过测试后确定稳定无Bug后再将这个程序的内容写入new_yolo.py
* `new_yolo.py`：Yolo节点代码，是经过测试后的稳定代码，也是一般比赛时使用的代码。到国赛时代码已经基本确定，new_yolo.py和new_yolo_d.py的内容是基本一样的。
* `test_llm_local.py`：本地部署大模型的节点代码。负责接收深度相机的RGB图像，向本地大模型发送请求，并将请求传回上位机。
* `test_qrcode.py`：二维码节点代码，负责识别二维码。

**其中以下部分是平时用于调试的：**
* `new_videologger.py`：视频调试代码，接收Resnet识别结果、Yolo节点的Yolo结果和PID计算的重要数值、Camera的图像，将这些输出为一个视频。视频上将Yolo识别到的物体框出，并标注PID参考值、注意点、PID计算的过程数值以及最终发布的速度。
* `test_carcontrol.py`：测试的代码，它会发布一个固定的速度和角速度信息到话题中，控制小车以固定速度运行。

**其中以下部分是没有被使用或者弃用的：**
* `car_keyboard.py`：使用CoBridge进行上位机和小车通信时，没有键盘控制，因此编写了这个代码，用于接收上位机发出的UDP数据包，其中包含上位机按下的按键，这个代码将其转换为对应的速度信息进行发布。代码是国赛时临时写的，最终没有真正使用。
* `depth_image.py`：深度相机有关代码，接收Yolo识别结果，并取障碍物面积最大的障碍物获取其与小车的距离。因为Yolo识别是使用单目摄像机而非深度相机，因此代码中使用了图像拉伸。但是由于深度相机和单目相机的视野不同因此距离获取并不准确，最终被弃用。
* `depth_logger.py`：深度相机有关代码，是最开始尝试使用深度相机的测试代码，接收深度相机的深度图识别结果，将其保存为一个视频输出。
* `depth_yolo.py`：深度相机有关代码，在学会如何使用深度相机时，姚俊臣尝试过使用深度相机+OpenCV识别障碍物的方法，但是效果不好，最后放弃了。
* `depth_yolo2.py`：与上面的代码相同，均被弃用。
* `new_pidctrl.py`：引入小车前进区域那两条直线后的旧PID控制代码，PID控制还处于Resnet节点中。
* `qrcode_wechat.py`：采用微信二维码识别的代码，但是效果与原来相比没有改善甚至有时更差，被弃用。
* `test_car_pid.py`：最初的PID控制代码、Resnet节点代码，后续经过了第一次的控制代码迭代。加入了以两条前行区域直线计算转向参考的代码。
* `test_llm.py`：云端火山大模型的节点代码，接收图像并向火山大模型API发送请求，在本地部署大模型后被弃用。
* `test_resnet.py`：*并非ROS2程序*，仅仅是调用摄像头与Resnet模型进行注意点识别，然后将识别结果输出为视频。在文件迁移的时候不知道被谁放进了ROS2的项目中。
* `test_yolo.py`：旧的Yolo节点代码，它接收Camera节点的Yolo识别结果，并选出面积最大的障碍物，将这个障碍物的信息发给Resnet节点用于避障的计算。


### 配置文件夹(`config`)
存储了以下文件：
* `config_car_keyboard.yaml`：对应键盘控制程序`car_keyboard.py`的配置，其中只有一项
    ```yaml
    udp_port: 8888      # UDP通信端口
    ```
* `config_image_compress.yaml`：对应压缩图像传输程序`image_comp.py`的配置。
    ```yaml
    scale_factor: 0.5       # 缩放比
    jpeg_quality: 50        # jpeg图像质量
    enable_compression: true        # 是否启用压缩图像
    ```
* `config_new_resnet.yaml`：对应Resnet节点`new_resnet.py`的配置。详细说明见自动控制程序部分。
* `config_new_yolo_d.yaml`：对应Yolo节点`new_yolo_d.py`或`new_yolo.py`的配置。详细说明见自动控制程序部分。
* `config_videologger.yaml`：对应调试程序`new_videologger.py`的配置。详细说明见自动控制程序部分。
* `config_yolo_plan.yaml`：在尝试开发路径规划程序时创建的，没有用处。

### 接口功能包(`test_interface`)
主要包括以下自定义接口：
* `ObjectMsg.msg`：障碍物矩形框的信息，用于给调试程序`new_videologger.py`提供障碍物信息
* `ObjPosition.msg`：旧的Yolo节点`test_yolo.py`使用的接口，传递面积最大障碍物的矩形框信息。
* `ResnetMsg.msg`：Resnet注意点识别结果信息
* `VideoLogMsg.msg`：包含了Yolo节点运行的一些重要信息，用于给调试程序`new_videologger.py`提供信息
